Perfecto 💪 — te dejo un to-do list claro y jerarquizado para mejorar la UX de tu flujo de creación de órdenes de servicio, eliminar el uso problemático del checkbox, y mantener sincronizado el render y el payload.

🧭 Objetivo general

Mejorar la experiencia de usuario y la coherencia de datos al crear órdenes de servicio, eliminando el uso del checkbox crearLinea y reemplazándolo por interacciones más naturales y predecibles.

✅ TO-DO LIST — Mejora UX + Lógica

1️⃣ Reemplazar el checkbox por controles más explícitos

🎯 Motivo:

El checkbox crearLinea confunde, ya que su cambio desmonta formularios y puede provocar pérdida de datos.

🧩 Acciones:

  • Eliminar el campo crearLinea del schema buildOrdenServicioFields.

  • Añadir un botón “➕ Agregar línea de servicio” debajo del formulario de línea actual. 

    jsx
    <Button
  variant="outlined"
  onClick={handleAgregarLinea}
>
  + Agregar línea de servicio
</Button>

  • Añadir un botón de eliminación por línea: 

    jsx
    <IconButton onClick={() => handleRemoveLinea(idx)} color="error">
  🗑️
</IconButton>

  • Opcionalmente, usar un Switch si se desea conservar la lógica de mostrar/ocultar una sección completa (pero sin desmontar datos guardados).

2️⃣ Mantener persistencia de datos al ocultar formularios

🎯 Motivo:

Si un usuario “desactiva” una línea o la oculta, no debería perder la información ya ingresada.

🧩 Acciones:

  • Cambiar el comportamiento actual de crearLinea: false para que no borre datos, sino que solo oculte la sección visualmente. 

    jsx
    if (field === 'crearLinea' && value === false) {
  return { ...prev, lineas: prev.lineas.map((l, i) => ({ ...l, visible: i === 0 })) };
}

  • Asegurar que buildPayload filtre solo las líneas visibles o activas: 

    js
    const lineasServicio = (orden.lineas || [])
  .filter(l => l.visible !== false)
  .map(({ tipoTrabajo, descripcion, precioUnitario, cantidad }) => ({
    tipoTrabajo,
    descripcion,
    precioUnitario,
    cantidad
  }));

3️⃣ Simplificar la lógica en el contexto

🎯 Motivo:

Actualmente handleChangeLinea mezcla lógica visual (montar/desmontar) con lógica de datos.

🧩 Acciones:

  • Extraer las operaciones de UI (mostrar/ocultar) en un hook separado (useLineasServicioUI).

  • Mantener en el contexto solo la manipulación de datos (addLinea, updateLinea, removeLinea).

  • Quitar el bloque que hace: 

    js
    if (field === 'crearLinea' && value === false) {
  updateOrden('lineas', []);
}

4️⃣ Ajustar buildPayload para soportar la nueva semántica

🎯 Motivo:

Ahora que ya no hay crearLinea, el backend debe recibir solo las líneas realmente activas.

🧩 Acciones:

  • Eliminar dependencia de orden.crearLinea.

  • Crear las líneas con: 

    js
    const lineasServicio = (orden.lineas || [])
  .filter(l => l.tipoTrabajo && l.descripcion)
  .map(l => ({
    tipoTrabajo: l.tipoTrabajo,
    descripcion: l.descripcion,
    cantidad: l.cantidad || 1,
    precioUnitario: l.precioUnitario || 0,
  }));

  • Validar que lineasServicio.length >= 1, y emitir error UX si no hay líneas válidas.

5️⃣ Feedback visual y confirmaciones

🎯 Motivo:

Evitar frustración cuando el usuario elimina accidentalmente una línea.

🧩 Acciones:

  • Al eliminar línea → mostrar confirmación tipo: 

    jsx
    confirm('¿Seguro que deseas eliminar esta línea? Los datos se perderán.')

  • Al enviar orden → mostrar resumen previo (modal o preview):

    • Cliente y equipo seleccionados

    • Total y número de líneas

    • Lista resumida de descripciones

  • Añadir toast notifications en éxito/error.

6️⃣ Mejorar nomenclatura

🎯 Motivo:

crearLinea induce a confusión; el naming debe reflejar su función real.

🧩 Acciones:

  • Renombrar crearLinea → agregarLinea (si se mantiene lógica booleana)

  • Renombrar lineasServicio → lineas (en el contexto) y mapear en payload según backend.

7️⃣ QA y pruebas

🎯 Motivo:

Evitar regresiones al eliminar lógica de control.

🧩 Acciones:

  • Probar flujo de creación con:

    • 0 líneas (debe rechazar)

    • 1 línea (debe aceptar)

    • 2 líneas, eliminando la segunda (debe aceptar)

  • Confirmar que el payload final solo contenga líneas visibles.

  • Confirmar que al recargar, las líneas previas se mantienen si no se completó envío.
